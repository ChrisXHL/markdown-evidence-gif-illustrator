# Visual Format Policy

默认优先 **高清静态图**，不是 GIF。

## Default rule
对每个配图点，先问：
- 能不能用 **一张高清静态图** 讲清楚？
- 能不能在不损失关键信息的情况下裁出合适比例？
- 是否必须强行限制成 16:9？如果会损失信息，就不要强制。

如果一张高清图可以讲清楚，**默认输出静态图**。
只有当一张图无法完整承载信息时，才退回到 GIF。

## Preferred format order
1. `static_hd_adaptive`
2. `static_hd_16_9`（仅在用户明确要求或信息适合时）
3. `gif`

## When to use static HD screenshots
优先使用静态图的情况：
- 定价表、排行榜、单个对比卡片
- 单屏可读的官方说明区域
- 一眼就能说明问题的图表或页面模块
- 公众号/内容平台对 GIF 支持差时

## Aspect ratio rule
- **信息完整性 > 固定比例**
- 如果 16:9 会裁掉关键数字、标题、来源标识，就改用自适应比例
- 只有在不损失关键内容时才使用 16:9

## HD requirements
静态图尽量满足：
- 直接从网页高分辨率截图，不从 GIF 或低清首帧二次放大
- 截图前确认页面真的加载完成：目标文本/模块可见、字体已渲染、必要时等待 lazy-load 内容稳定
- 尽量保留标题 / 来源标识 / 关键数字 / 关键说明文案
- 对关键证据句、关键词、数据行、模块边界，优先加清晰高亮/指示，而不是裸截图
- 高亮应尽量不遮挡文字：文本优先用荧光笔/下划线式标注，模块或 UI 区域再用边框式标注
- 图内文字标签默认不要加；除非用户明确要求，否则只保留高亮，不叠加说明牌
- 小红点/鼠标点默认也不要加，除非用户明确要求强调指向
- 避免无意义空白与无关装饰
- 优先保证可读性，而不是统一外观
- 比例规则默认以 16:9 为基础；若需要更高画幅以容纳完整信息，可放宽到 16:x，但 x 不得超过 16（即不超过 1:1）
- 如全页截图会削弱重点，优先截取“证据区域 + 少量上下文”
- 遇到登录墙、订阅墙、社交弹层等导致证据不可读时，不要硬截；应更换为更可用且 provenance 更好的页面

## When to fallback to GIF
只有满足以下任一，才建议 GIF：
- 关键信息跨多屏，单张截图无法完整表达
- 需要通过滚动才能从标题走到关键证据位置
- 需要展示“先定位来源，再看到关键区域”的过程
- 页面是动态展开/切换后才出现关键内容

## Asset naming
建议：
- 高清静态图放在 `assets/images/`
- 动图放在 `assets/gif/`

例如：
- `assets/images/01-pricing-source.png`
- `assets/gif/02-benchmark-source.gif`

## Markdown insertion
静态图和 GIF 都按普通 Markdown 图片语法插入。
区别只在文件路径和配图说明，不在插入方式。
